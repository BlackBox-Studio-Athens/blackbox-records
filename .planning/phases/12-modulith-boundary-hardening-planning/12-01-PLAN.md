# 12-01 Plan - Define Module Model And Verification Design

## Objective

Lock the explicit module map, one repo boundary manifest, root-first entrypoint policy, and future TypeScript-native
verification approach for the modulith-style hardening milestone.

## Scope

- define the exact path-based ownership map per module
- define the allowed-dependency matrix between modules
- define one repo machine-readable manifest with directory-first ownership and explicit exceptions
- define future verifier surfaces for no cycles, allowed dependencies, public-API-only access, and unknown-file drift
- keep the design rooted in `eslint-plugin-boundaries`, `dependency-cruiser`, and explicit facades instead of heavier
  framework adoption

## Tasks

<task id="12-01-01" title="Lock the manifest and ownership map">
  <action>
    Translate the approved module names into explicit owned paths, allowed dependencies, entrypoint policy, and
    documented exceptions inside one repo manifest plus the human-readable module docs. Record any path that is still
    ambiguous as a split target instead of silently giving it shared ownership.
  </action>
  <acceptance_criteria>
    The manifest and module canvases together name concrete owned files or directories, and no critical hotspot is left
    unowned.
  </acceptance_criteria>
</task>

<task id="12-01-02" title="Define root-first entrypoints and named interfaces">
  <action>
    Decide which modules expose only a root facade and which need extra named interfaces such as `api.ts` or `spi.ts`.
    Lock the rule that temporary compatibility facades are disallowed and that deep-import paths are not a substitute
    for named interfaces.
  </action>
  <acceptance_criteria>
    The future refactor can tell whether an import should target a root facade, a named interface, or be rejected.
  </acceptance_criteria>
</task>

<task id="12-01-03" title="Define the ESLint boundary layer">
  <action>
    Plan `eslint-plugin-boundaries` as the primary import-boundary mechanism. Prefer the `boundaries/dependencies` rule
    over legacy rule shapes, plan strict fail-fast coverage for covered module roots, and define how `no-unknown-files`
    or equivalent strict classification will stop stray files from appearing inside the hardened area.
  </action>
  <acceptance_criteria>
    The plan names how imports, entrypoints, and unclassified files will be rejected inside the covered module roots.
  </acceptance_criteria>
</task>

<task id="12-01-04" title="Define the dependency graph layer">
  <action>
    Plan `dependency-cruiser` for cycle checks and folder-level dependency rules. Use directory-first manifest data,
    error severities, and group-matching capable rules so the future checks can reject cycles and forbidden
    cross-module folder dependencies.
  </action>
  <acceptance_criteria>
    The plan names how the future refactor will reject cycles and illegal folder-level dependencies rather than only
    relying on file-by-file lint errors.
  </acceptance_criteria>
</task>
