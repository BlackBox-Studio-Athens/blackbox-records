# 12-39 Plan - Test Performance Analysis

## Outcome

Measure the current test and validation loop, then decide whether Phase 12 should change the test topology for faster
module refactors without adding avoidable complexity.

## Scope

- Measure current timings for:
  - focused app-shell seam tests;
  - `pnpm test:app-shell`;
  - package-level Vitest runs;
  - `pnpm test:unit`;
  - test-adjacent checks that materially affect the Phase 12 feedback loop.
- Use native ecosystem capabilities only for measurement and candidate changes:
  - Vitest CLI/config/reporters/pools/projects/sharding options;
  - pnpm workspace filtering, recursive execution, and parallelism;
  - TypeScript project references only if they simplify and speed existing type-aware tooling;
  - Astro check behavior as-is unless a native Astro/TypeScript configuration improvement is clearly supported.
- Do not add custom benchmark scripts or one-off local profiling tools.
- Do not add Nx in this slice; it remains deferred by user decision.
- Consider a new dependency only if it is well maintained, robust, fits the existing pnpm/TypeScript/Astro stack, and
  provides clear value beyond native Vitest/pnpm/TypeScript capabilities.

## Decision Questions

1. Is the current Phase 12 fast loop already sufficient with focused Vitest commands plus `pnpm test:app-shell`?
2. Can `pnpm test:unit` safely run package test suites in parallel without making output or failures harder to inspect?
3. Would Vitest projects, workspace configs, sharding, or pool tuning reduce total runtime without obscuring module
   ownership?
4. Would TypeScript project references improve lint/typecheck performance enough to justify a repo-wide config change?
5. Is any external task runner or cache library justified now, or should the repo stay on native pnpm/Vitest commands?

## Deliverables

- A short evidence table of measured timings.
- A recommendation: keep current loop, apply a native configuration change, or open a later tooling slice.
- If an improvement is approved within this slice, keep it small, native-first, and verified by the same required gates.

## Verification

- Measurement commands and output summaries recorded in the summary artifact.
- If config changes are made:
  - focused affected test command;
  - `pnpm test:unit`;
  - `pnpm check`;
  - `pnpm build` if the change can affect build/test resolution.
