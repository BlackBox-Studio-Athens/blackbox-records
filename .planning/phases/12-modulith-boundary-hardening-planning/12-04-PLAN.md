# 12-04 Plan - Plan Legacy-Open Governance And Branch Workflow

## Objective

Define the future closure path for `cms-admin`, the hard-closure policy for legacy-open modules, and the governance
rules that keep the refactor reviewable without changing repo automation settings yet.

## Scope

- `cms-admin` module closure planning
- machine-readable manifest governance
- future compact repo-local `AGENTS.md` rule set
- branch workflow and slice sizing rules for the execution milestone

## Files

- `apps/web/src/lib/admin/decap-config.ts`
- `apps/web/src/pages/admin/**`
- `.planning/adrs/ADR-004-modulith-style-module-boundaries.md`
- `.planning/codebase/MODULES.md`
- `.planning/codebase/module-boundaries.manifest.json`
- `AGENTS.md`

## Tasks

<task id="12-04-01" title="Plan cms-admin closure under a hard exception bar">
  <action>
    Define the future split between admin route surfaces, Decap YAML generation, content-schema awareness, and hosted
    auth or base-path handling so `cms-admin` can stop being a large mixed hotspot. Do not allow new temporary-open
    modules as a convenience escape hatch.
  </action>
  <acceptance_criteria>
    The `cms-admin` canvas names why it is temporary-open, what exact exit criteria will close it later, and what
    refactor moves are still forbidden while it remains open-temporary.
  </acceptance_criteria>
</task>

<task id="12-04-02" title="Plan manifest and documentation governance">
  <action>
    Define how the manifest, `MODULES.md`, and per-module canvases stay in sync. Tooling should read the manifest,
    while human review should rely on the Markdown docs. Future slices must update both in the same change when module
    ownership, entrypoints, or allowed dependencies move.
  </action>
  <acceptance_criteria>
    The future governance rule set is explicit about which file is for tooling, which files are for human review, and
    how drift between them is prevented.
  </acceptance_criteria>
</task>

<task id="12-04-03" title="Plan manual branch workflow">
  <action>
    Lock the future branch policy to one approved execution slice per branch while keeping branch creation and
    execution sequencing manual. Do not change `.planning/config.json` automation settings in this phase.
  </action>
  <acceptance_criteria>
    The future execution milestone has explicit branch and review sizing rules without enabling parallelization or
    automatic worktree behavior.
  </acceptance_criteria>
</task>
