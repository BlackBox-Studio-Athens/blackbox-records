# 12-02 Plan - Plan Web Shell Decomposition

## Objective

Prepare the future decomposition of `app-shell`, `player`, and `store-cart` so `AppShellRoot` becomes a thin
composition root over smaller modules with stable contracts, explicit entrypoints, and localized tests.

## Scope

- `app-shell`
- `player`
- `store-cart`
- affected `checkout-web` touchpoints only where they currently cross into cart or shell orchestration

## Files

- `apps/web/src/components/app-shell/AppShell.astro`
- `apps/web/src/components/app-shell/AppShellRoot.tsx`
- `apps/web/src/components/app-shell/player-*.ts`
- `apps/web/src/components/music/MusicStreamingServiceListenTrigger.astro`
- `apps/web/src/lib/app-shell/routing.ts`
- `apps/web/src/lib/store-cart.ts`
- `apps/web/src/components/store/StoreCartButton.tsx`
- `apps/web/src/components/store/StoreCartDrawer.tsx`
- `apps/web/src/components/store/StoreItemPurchaseActions.tsx`
- `apps/web/src/components/store/CheckoutOrderSummary.tsx`

## Compatibility Invariants

- top-level shell section navigation behavior stays unchanged
- overlay behavior stays unchanged
- player open, minimize, reopen, and stop behavior stays unchanged
- StoreCart remains browser convenience state only
- checkout still uses the current browser-safe public API flow
- callers move to new root entrypoints inside the same slice; no temporary compatibility facades

## Tasks

<task id="12-02-01" title="Characterize shell behavior and relocate tests">
  <action>
    Define the helper-level and Browser Use checks that must exist before moving shell logic. Treat routing, overlay,
    history, scroll or focus reset, player continuity, and cart drawer orchestration as separate compatibility
    concerns. Plan the touched test suites to move closer to the hardened modules instead of keeping all logic guarded
    only by distant broad tests.
  </action>
  <acceptance_criteria>
    The future implementation can add or relocate tests before code motion instead of rewriting the shell blind.
  </acceptance_criteria>
</task>

<task id="12-02-02" title="Define target internal modules and root entrypoints">
  <action>
    Split the future shell work into explicit internal seams: shell page snapshot logic, overlay fetch and cache,
    shell history integration, player session coordination, and StoreCart browser bridge. Record the target root
    entrypoint for each extracted module and keep named interfaces exceptional rather than default.
  </action>
  <acceptance_criteria>
    `AppShellRoot` has a documented target shape as a thin composition root rather than a multi-concern owner, and the
    new seams have obvious import targets.
  </acceptance_criteria>
</task>

<task id="12-02-03" title="Lock event usage and the cart boundary with checkout-web">
  <action>
    Lock the future seam between pure StoreCart state, cart event transport, cart UI, and checkout-web behavior so
    add-to-cart and checkout-return interactions do not keep leaking orchestration back into the shell. Keep browser
    events only where they remain the least-wrong decoupling tool, not as a blanket pattern.
  </action>
  <acceptance_criteria>
    The future shell refactor distinguishes direct APIs from exceptional event seams and does not create new vague
    cross-module browser orchestration paths.
  </acceptance_criteria>
</task>
