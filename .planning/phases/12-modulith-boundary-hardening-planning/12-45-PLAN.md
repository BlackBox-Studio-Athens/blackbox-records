# 12-45 Plan - Test Performance Follow-Up Analysis

## Outcome

Run a second, immediate Phase 12 test-performance analysis before more refactor slices, focused on whether the current
module-migrated repo can make tests and test-adjacent checks faster without introducing custom tooling or unnecessary
topology.

## Scope

- Re-measure the current feedback loop after the latest app-shell and `cms-admin` extractions:
  - focused Vitest commands for touched modules;
  - package-level Vitest commands;
  - root `pnpm test:unit`;
  - `pnpm check` sub-steps that materially affect refactor feedback;
  - the boundary gates that Phase 12 relies on.
- Investigate the TypeScript/ESLint multiple-project warning path and decide whether to:
  - keep the current `noWarnOnMultipleProjects` suppression because it is only noise;
  - move to a single root TypeScript solution config with `references` if it speeds type-aware tooling without making
    package ownership harder to understand;
  - use a native, maintained TypeScript ESLint project-service option if it fits the installed ecosystem better than
    manual project lists;
  - leave the warning suppression in place and open a later tooling slice if evidence is inconclusive.
- Use only native ecosystem capabilities or well-maintained libraries that add clear value:
  - pnpm workspace filtering, recursion, and parallel execution;
  - Vitest CLI/config/reporters/projects/pools/sharding when justified by measurements;
  - TypeScript project references or solution-style configs when they simplify and speed the repo;
  - official Astro/TypeScript/ESLint configuration options.
- Do not add custom benchmark scripts, bespoke timing wrappers, ad hoc shell utilities, or a task-runner/cache framework in
  this slice.
- Keep Nx and formal submodule modeling deferred unless this analysis produces new evidence strong enough for a separate
  human decision.

## Decision Questions

1. Is the current `pnpm test:unit` parallel package gate still the fastest simple root unit gate after the newer module
   splits?
2. Are any focused module commands missing from the documented Phase 12 fast loop?
3. Is `pnpm check` slowed primarily by type-aware ESLint, Astro check, dependency-cruiser, or formatting, and is there a
   low-complexity native improvement?
4. Does the multiple-project warning indicate real work duplication, or is suppressing it still the least complex choice?
5. Would TypeScript `references`, TypeScript ESLint project service, Vitest projects, or Vitest pool tuning improve
   feedback enough to justify their maintenance cost?
6. Should any external maintained library be added now, or should the repo stay on pnpm/Vitest/TypeScript/Astro native
   capabilities?

## Deliverables

- A new `12-45-SUMMARY.md` with timing evidence, warning analysis, and a decision.
- Any accepted configuration change kept minimal and reversible.
- Updated Phase 12 docs if the recommended fast-loop commands or warnings policy changes.

## Verification

- Record measured commands and observed durations in the summary artifact.
- If no config changes are made, verify with the measurement commands only.
- If config changes are made, run the smallest affected command first, then:
  - `pnpm test:unit`;
  - `pnpm check`;
  - `pnpm build` if resolution, package scripts, TypeScript config, or Astro check behavior can be affected.
