# 12-03 Plan - Plan Backend Commerce Boundary Hardening

## Objective

Prepare the future split between `public-commerce-http`, `checkout-core`, `orders`, `stock`, and `operator-stock`
without changing the public checkout behavior first and while moving the backend toward explicit root entrypoints plus
fail-fast boundary checks.

## Scope

- public HTTP routes and contracts
- application commerce modules
- internal operator HTTP and client surfaces
- generated public and internal client ownership

- `apps/backend/src/interfaces/http/routes/register-public-commerce-routes.ts`

## Tasks

<task id="12-03-01" title="Separate HTTP contract, route, and composition concerns">
  <action>
    Define the future split between schemas, route definitions, handlers, and service composition so route files stop
    wiring infrastructure directly beside the contract surface. The target state should leave one obvious root
    entrypoint per backend module rather than many route-level import targets.
  </action>
  <acceptance_criteria>
    The target ownership for route registration, error mapping, return-origin policy, and infrastructure assembly is
    documented and reviewable.
  </acceptance_criteria>
</task>

<task id="12-03-02" title="Lock backend root entrypoints and named interfaces">
  <action>
    Decide which backend surfaces become root APIs and which become named interfaces: public contracts, internal
    contracts, application use-case indexes, repository SPIs, and generated clients. Reuse package `exports` where a
    real package already exists, but do not require package extraction just to get boundary clarity.
  </action>
  <acceptance_criteria>
    Future backend refactors can preserve public and internal contracts while moving implementation detail behind
    explicit import targets.
  </acceptance_criteria>
</task>

<task id="12-03-03" title="Define verification and migration order">
  <action>
    Plan the order for future backend hardening: route characterization tests first, touched test relocation second,
    contract preservation third, composition extraction fourth, and strict boundary enforcement once the root
    entrypoints exist. Do not plan temporary compatibility facades for old route-internal imports.
  </action>
  <acceptance_criteria>
    The future execution slices do not start by moving code before they have route-level and application-level safety
    nets, and the resulting module boundaries can be enforced mechanically.
  </acceptance_criteria>
</task>
