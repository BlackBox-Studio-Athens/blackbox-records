# 12-57 Test Feedback Loop Performance Analysis

## Summary

Re-measure the Phase 12 test and check feedback loop after the app-shell and cms-admin module migrations, then decide whether the repo can improve test speed without adding custom scripts or unnecessary tooling complexity.

This is an immediate Phase 12 tooling slice before deeper refactors continue. It must preserve the existing module boundary strategy, avoid Nx/submodule adoption unless a separate human decision reopens that topic, and keep `eslint-plugin-boundaries` as part of the required boundary gate.

## Scope

- Measure the current fast and full feedback loops with native ecosystem commands:
  - focused Vitest package/module commands for recently migrated modules;
  - `pnpm test:unit`;
  - `pnpm check`;
  - `pnpm build` only as a baseline when config changes could affect build resolution.
- Identify the slowest useful gates and distinguish test runtime from lint/type/content/boundary runtime.
- Evaluate only low-complexity improvements from native or well-maintained ecosystem surfaces:
  - pnpm workspace filtering, recursion, and parallel execution;
  - Vitest CLI/config/reporters/projects/pools/sharding when justified by evidence;
  - TypeScript project references or solution-style configs if they reduce repeated work without creating config sprawl;
  - official Astro, TypeScript, ESLint, `typescript-eslint`, and `eslint-plugin-boundaries` options;
  - a maintained library only if it is robust, actively maintained, and clearly adds value over the current stack.
- Decide whether to keep the current loop, apply a small native configuration change, or open a later tooling slice.

## Out Of Scope

- No custom benchmark scripts, timing wrappers, ad hoc shell utilities, or bespoke local profiling code.
- No Nx adoption in this slice.
- No formal submodule model adoption in this slice.
- No weakening of `pnpm audit:module-boundaries`, `pnpm depcruise:boundaries`, `pnpm check:boundaries`, or `eslint-plugin-boundaries` enforcement.
- No test deletion or coverage narrowing solely to improve speed.

## Decision Questions

1. After the recent app-shell and cms-admin migrations, is `pnpm test:unit` still using the fastest simple package-level shape?
2. Are focused module commands documented well enough for Phase 12 refactor work, or should package scripts expose a better fast path?
3. Is test runtime actually the bottleneck, or are type-aware linting, Astro check, dependency-cruiser, or boundary audits now more expensive?
4. Would TypeScript `references`, Vitest projects/pool tuning, or `typescript-eslint` project-service settings improve speed enough to justify their maintenance cost?
5. Is any external maintained library justified now, or should the repo stay on native pnpm/Vitest/TypeScript/Astro/ESLint capabilities?

## Deliverables

- A `12-57-SUMMARY.md` with measured commands, approximate timings, bottlenecks, and the decision.
- Any accepted configuration/package-script change kept minimal, documented, and reversible.
- Updated Phase 12 docs if the approved fast-loop command set changes.

## Verification

- Record measured commands and timing evidence in the summary artifact.
- If no config changes are made, verify with the measurement commands and planning diff only.
- If config or package-script changes are made, run the smallest affected command first, then:
  - `pnpm test:unit`;
  - `pnpm check`;
  - `pnpm build` when test/check/build resolution or TypeScript/Astro behavior can be affected.
