# 17-01 Plan - Add Centralized Slugify-backed Slug Tooling

## Objective

Adopt `@sindresorhus/slugify` for slug generation and validation through a BlackBox-owned wrapper while preserving existing slug identities.

## Tasks

1. Add the dependency.
   - Install `@sindresorhus/slugify` in the package that owns the slug wrapper.
   - Use latest-compatible version at implementation time; latest checked on 2026-05-22 was `3.0.0`.

2. Add the wrapper.
   - Create explicit functions for generating slug suggestions and validating existing slugs.
   - Keep uniqueness/collision behavior explicit.
   - Avoid direct `slugify` imports outside the wrapper.

3. Migrate generation call sites.
   - Start with admin/CMS helpers and script-side fallback slug generation.
   - Leave existing canonical slug data unchanged.
   - Add an audit helper or tests that prove known slugs remain stable.

4. Add tests.
   - Accented text, punctuation, symbols, casing, whitespace.
   - Collision reporting.
   - Store item slug stability and legacy release ID separation.

5. Record validation evidence in `17-VALIDATION.md`.

## Verification

- Targeted slug utility/admin/catalog/script tests.
- `pnpm test:unit`
- `pnpm check`
- `pnpm build`

## Stop Conditions

- Stop before renaming committed content or routes.
- Stop if a generated slug differs from a canonical committed slug and no compatibility path exists.
- Stop before changing commerce identity mappings.
