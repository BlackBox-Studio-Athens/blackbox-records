# Phase 06.1.1 Research - Internal Stock Operations And Operator Access

## Research Goal

Define the safest thin operator model for shared stock reconciliation in a dual-deploy Astro + Worker + D1 architecture, with Google-backed staff auth and no spreadsheet-as-truth drift.

## Findings

### 1. Separate protected hostname is the cleanest staff boundary

- Cloudflare currently recommends Custom Domains when a Worker is the application's origin, and recommends production Workers run on a route or custom domain instead of only `workers.dev`.
- For this project, a separate protected hostname is cleaner than mixing public and staff endpoints under one hostname or one path tree.
- Phase contract:
  - public storefront hostname remains public
  - internal stock hostname is separately protected by Cloudflare Access

### 2. Cloudflare Access + Google fits staff-only stock operations

- Cloudflare currently supports Google as an identity provider for Access.
- Access policies can allow or deny access by email or group-based rules, which is enough for a small label team.
- This avoids building a custom auth system and avoids trying to reuse Decap's Git-backed auth for runtime D1 writes.

### 3. Operator identity can be captured at the Worker boundary

- Access-protected HTTP applications can expose authenticated user identity information to the protected origin flow.
- The Worker can use the authenticated email identity supplied by the Access-protected request flow as `actor_email`.
- Every `StockChange` and `StockCount` should also record `recorded_at` at write time.

### 4. D1 should remain the only stock authority

- The stock ledger is now:
  - `Variant`
  - `Stock`
  - `StockChange`
  - `StockCount`
- This is not a full ERP; it is the minimum honest ledger for online plus offline sales.
- Spreadsheets can still exist as temporary event capture or exports, but they must never become a second live source of truth.

### 5. Offline retail pressure justifies a conservative online stock field

- Bazaar and event sales can be delayed or imperfectly recorded.
- Relying only on total physical stock makes online oversell too easy.
- The stock model should therefore include:
  - total `Stock`
  - conservative `OnlineStock`
- Public storefront and checkout flows should consume `OnlineStock`, not raw stock-balance details.

### 6. Internal API should be small and explicit

Recommended internal API surface:

- `GET /api/internal/variants`
  - search/list `Variant` records with enough context to identify the item
- `GET /api/internal/variants/:id/stock`
  - returns current `Stock`, `OnlineStock`, and recent stock history
- `POST /api/internal/stock-changes`
  - appends a `StockChange` row and recomputes balance
- `POST /api/internal/stock-counts`
  - records a `StockCount` row and reconciles balance

### 7. Thin UI beats spreadsheet-first or full admin in v1

- The first operator UI only needs:
  - search/select a `Variant`
  - inspect stock summary and recent history
  - submit a `StockChange`
  - submit a `StockCount`
- This is operationally sufficient for low-volume online sales plus offline events.
- A full order-management dashboard remains out of scope.

## Risks To Avoid

- Reusing Decap auth for runtime stock writes
- Treating spreadsheets and D1 as dual authorities
- Exposing stock-write APIs on the public shopper hostname
- Making public checkout depend on raw physical stock instead of a conservative online quantity
- Expanding this phase into a full ERP/OMS

## Recommended Contract

- Staff auth: Cloudflare Access + Google
- Staff boundary: separate protected backend hostname
- Ledger authority: D1 only
- Stock language: `Variant`, `Stock`, `StockChange`, `StockCount`
- Browser contract:
  - public storefront sees `StoreItem`, `ItemAvailability`, and `OnlineStock`
  - staff UI sees stock details and write actions through protected internal APIs
