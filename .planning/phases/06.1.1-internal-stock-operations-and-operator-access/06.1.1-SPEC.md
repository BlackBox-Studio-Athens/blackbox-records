# Phase 06.1.1 SPEC - Internal Stock Operations And Operator Access

## Deliverable

Produce a decision-complete plan for a protected staff-only stock operations surface in the Worker backend, including operator auth, stock-write APIs, a thin internal UI, and spreadsheet boundary policy.

## Falsifiable Requirements

1. Internal stock operations are protected by Google-backed Cloudflare Access on a separate backend hostname and do not require shopper login.
2. The stock ledger vocabulary explicitly uses `Variant`, `StockBalance`, `StockChange`, and `StockCount`.
3. The Worker backend exposes internal APIs for variant discovery, stock visibility, `StockChange`, and `StockCount`.
4. Every internal stock write records operator attribution through `actor_email` and `recorded_at`.
5. The operator UI supports searching variants, reading current stock, recording `StockChange`, recording `StockCount`, and reviewing recent stock history.
6. The planning docs explicitly state that D1 is authoritative and spreadsheets are temporary capture/reporting tools only.
7. The stock model explicitly includes a conservative `online_available` quantity separate from total `StockBalance`.
8. The internal stock APIs are implemented as TypeScript Hono routes with business-named use cases and mandatory backend tests.

## Boundaries

- This phase does not create a full OMS or warehouse system.
- This phase does not add shopper-facing login.
- This phase does not move editorial content editing into the Worker backend.
- This phase does not change Stripe checkout semantics.

## Acceptance

- A human can read the resulting docs and implement protected stock tooling for label staff without re-deciding auth, API shape, stock terminology, or spreadsheet policy.
