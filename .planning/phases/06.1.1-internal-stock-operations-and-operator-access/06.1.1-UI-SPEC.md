# Phase 06.1.1 - UI Design Contract

## Experience Principles

1. This is an internal tool, not a brand-expression surface.
2. Clarity and safety matter more than style.
3. Current stock, online-available stock, and write actions must be visually separated.
4. Operator attribution and recent history must be visible at the moment of change.
5. Reconciliation after an offline event must feel fast, not bureaucratic.

## Route Contract

- Protected operator hostname:
  - `/stock/` - static Astro search, overview, detail, history, and write actions
  - `/stock/?variantId=<variantId>` - shareable detail state for dynamic backend variants
- Protected API namespace on the same hostname:
  - `/api/internal/variants`
  - `/api/internal/variants/:variantId/stock`
  - `/api/internal/variants/:variantId/stock/history`
  - `/api/internal/variants/:variantId/stock/changes`
  - `/api/internal/variants/:variantId/stock/counts`

## Stock Overview

- Primary layout is desktop-first with:
  - search input
  - result table or list
  - quick stock summary per row
- Each result row includes:
  - title
  - source type (`release` or `distro`)
  - subtitle (`artist` or label metadata)
  - format / distinguishing variant label
  - current `Stock`
  - current `OnlineStock`
  - last updated time
  - `Open Stock` action

## Variant Detail View

- Above the fold:
  - title and variant label
  - current `Stock`
  - current `OnlineStock`
  - last recorded change metadata
- Main body is split into two functional blocks:
  1. `StockChange` form
  2. `StockCount` form
- Recent history appears beside or below the forms, depending on viewport width.

## Write Forms

### `StockChange`
- Fields:
  - quantity delta
  - reason
  - optional channel
  - optional notes
- Examples of reasons include:
  - offline sale
  - restock
  - correction
  - damage/loss
  - return

### `StockCount`
- Fields:
  - counted quantity
  - optional notes
- This path is visually framed as a reconciliation action, not a normal sale entry.

## History Contract

- Recent history shows:
  - action type (`StockChange` or `StockCount`)
  - effective quantity change or counted value
  - reason / notes
  - `actor_email`
  - `recorded_at`
- History must make it obvious who changed stock and when.

## Copy Contract

- Tone is direct and operational.
- Use domain terms exactly:
  - `Variant`
  - `Stock`
  - `OnlineStock`
  - `StockChange`
- `StockCount`
- Avoid ecommerce marketing language.

## Implementation Contract

- Astro owns the stock UI; the Worker must not serve stock HTML routes.
- The UI calls same-origin `/api/internal/*` on the protected operator hostname.
- `PUBLIC_BACKEND_BASE_URL` is only a local split-port fallback for static web -> local Worker development.
- The browser never sends `actor_email`; backend writes derive it from Cloudflare Access.

## Non-Goals

- No public-brand visual polish requirement
- No customer-facing login
- No order-management dashboard
- No spreadsheet sync UI in this phase
