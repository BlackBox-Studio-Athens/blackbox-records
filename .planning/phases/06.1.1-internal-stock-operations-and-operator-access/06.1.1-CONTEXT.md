# Phase 06.1.1: Internal Stock Operations And Operator Access - Context

**Gathered:** 2026-04-20
**Status:** In progress

<domain>
## Phase Boundary

Phase 06.1.1 adds the protected operator-facing stock workflow on top of the Worker backend and D1 foundation. It covers staff auth, internal stock APIs, a thin stock UI, and spreadsheet policy before public checkout consumes live stock.

</domain>

<decisions>
## Implementation Decisions

### Auth boundary
- **D-01:** Internal stock operations use Google-backed Cloudflare Access, not Decap auth reuse.
- **D-02:** The stock tool lives on a separate protected backend hostname rather than on a mixed public/staff path.
- **D-03:** Public storefront and shopper checkout remain unauthenticated in this phase.
- **D-13:** Repo docs refer to the protected operator hostname as `ops.<managed-zone>` until the real account-owned custom domain is provisioned; this hostname stays separate from the public sandbox `workers.dev` backend.
- **D-14:** Access policy for the operator hostname uses Google plus an explicit operator email allowlist; actual allowlisted addresses stay out of the repo.
- **D-15:** Worker-side operator attribution is extracted from the Access-authenticated request header `cf-access-authenticated-user-email` and later persisted as `actor_email`.

### Stock ledger language
- **D-04:** The stock model uses `StoreItem` for storefront identity and `Variant` for the actual sellable unit.
- **D-05:** D1 is the authoritative ledger for `Stock`, `StockChange`, and `StockCount`.
- **D-06:** Each `Variant` also tracks a conservative `OnlineStock` quantity so offline reconciliation drift does not automatically expose full physical stock online.

### Internal API contract
- **D-07:** Protected internal APIs cover variant lookup, current stock visibility, stock history, `StockChange`, and `StockCount`.
- **D-08:** Every stock write records `actor_email`, `recorded_at`, `reason`, and optional notes.
- **D-09:** Shopper-facing APIs and checkout endpoints remain separate from staff-only stock-write APIs.

### Operator workflow
- **D-10:** The operator UI is intentionally utilitarian and optimized for fast correction, not for public-brand presentation.
- **D-11:** Staff must be able to search a `Variant`, inspect current `Stock` plus `OnlineStock`, submit a `StockChange`, submit a `StockCount`, and review recent history.
- **D-12:** Spreadsheets remain temporary capture/reporting tools only; they may inform reconciliation but never become the live stock source of truth.

### the agent's Discretion
- Exact component structure of the internal stock UI
- Whether stock history is returned inline from the stock endpoint or by a dedicated read endpoint
- Whether the earliest sandbox version uses a protected custom domain immediately or an Access-protected temporary hostname during setup

</decisions>

<specifics>
## Specific Ideas

- After offline events, staff should be able to reconcile either by entering one or more `StockChange` rows or by entering a `StockCount` if the exact event sales are unknown.
- The public storefront should only ever consume `OnlineStock`, never raw staff reconciliation fields.
- The first internal stock UI can be desktop-first and plain; it only needs to be safe, fast, and understandable to the label team.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/pages/admin/index.astro`: existing staff-facing entry area in the repo, useful as a contrast point for why stock operations are a separate tool and auth boundary
- `src/pages/shop/index.astro`: compatibility redirect baseline that the internal tool must not disturb

### Established Patterns
- Decap in this repo is Git-backed and production-authenticated through DecapBridge PKCE; it is not a runtime D1 write path.
- The app shell and public site remain static Astro surfaces; internal stock operations belong behind the protected operator hostname, with Astro owning UI and the Worker owning `/api/internal/*`.

### Integration Points
- Worker backend from Phase 5 provides the runtime host and secrets boundary.
- D1 + Prisma from Phase 6.1 provide the stock ledger storage layer that this phase operationalizes for staff.
- `apps/backend/src/interfaces/http/auth/operator-identity.ts` is the first typed seam for Access-derived operator identity on protected internal routes.
- `apps/backend/src/interfaces/http/routes/register-internal-stock-routes.ts` now owns the internal stock OpenAPI surface under `/api/internal/variants/*`.
- `apps/web/src/pages/stock/index.astro` now owns the protected static stock operations UI at `/stock/`, with `/stock/?variantId=<variantId>` as the detail state.
- The protected operator hostname routes static Astro `/stock/` UI and Worker-owned `/api/internal/*` behind Cloudflare Access.
- `apps/backend/src/application/commerce/stock/` now owns the application-layer stock read/write use cases on top of the D1 repositories.

</code_context>
