# Phase 5.1 Research - Commerce Domain Architecture And Source-Of-Truth Research

## Standard Stack

- Astro content collections for editorial content and static page generation
- separate Cloudflare Worker backend as BFF/API surface
- Stripe Products/Prices and Checkout for sellable commerce data and payment flow
- D1 plus Prisma runtime access for operational state and mappings

## Architecture Patterns

### Why the projection layer is required

- `releases` already model editorial release content and artist relationships.
- `distro` entries already model editorial card content and currently still carry `fourthwall_url`.
- Those two content sources are useful storefront inputs, but neither of them is the right place to hold live sellable state, stock, or Stripe identity.
- The native store therefore needs a projection layer that derives a stable storefront-facing `CatalogItem` from editorial content without turning Astro content into the commerce authority.

### Required implementation posture

- The browser and static storefront routes should consume `CatalogItem` and later `VariantSnapshot`, not Stripe-native or D1-native records.
- The Worker backend is the only layer allowed to translate between storefront contracts, Stripe sellable data, and D1 mappings.
- Editorial content stays useful and minimal when it remains descriptive instead of trying to encode live stock or payment state.
- `CatalogItem` should remain stable even if backend variant mappings or Stripe configuration change underneath it.

## Don't Hand-Roll

- Do not duplicate editorial content into D1.
- Do not use Stripe as the home for editorial media or prose.
- Do not expose direct browser writes to D1 or Stripe secret APIs.
- Do not treat the current `fourthwall_url` field as the future native shop identity.

## Common Pitfalls

- collapsing `CatalogItem` and `Variant` into one object too early
- storing pricing, stock, Stripe IDs, or checkout-owned fields in Astro collections
- letting the frontend consume Stripe-specific IDs or D1 record identifiers directly
- coupling browse/product-detail pages directly to backend persistence shapes instead of a storefront contract
- treating the projection layer as if it were a new permanent source of truth instead of a derived view

## Downstream implementation guidance

- static frontend consumes `CatalogItem` for browse/PDP
- frontend fetches `VariantSnapshot` from temporary adapter or Worker API without changing UI contract
- backend repositories hide D1 and Stripe implementation details behind interfaces

## Recommendation

Implement the storefront around a stable `CatalogItem` plus `VariantSnapshot` contract. Keep the browser coupled only to those contracts and route slugs. Put Stripe IDs, inventory values, and order state behind the Worker backend and D1 mappings.
