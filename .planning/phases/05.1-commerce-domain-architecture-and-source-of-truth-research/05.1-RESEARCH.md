# Phase 5.1 Research - Commerce Domain Architecture And Source-Of-Truth Research

## Standard Stack

- Astro content collections for editorial content and static page generation
- separate Cloudflare Worker backend as BFF/API surface
- Stripe Products/Prices and Checkout for sellable commerce data and payment flow
- D1 plus Prisma runtime access for operational state and mappings

## Architecture Patterns

### Recommended domain split
- `Artist`: editorial only
- `Release`: editorial content that may expose shop links
- `DistroEntry`: editorial/catalog content that may expose shop links
- `CatalogItem`: storefront-facing projection derived from one content source
- `Variant`: sellable unit used by checkout, stock, and Stripe mapping

### Recommended source-of-truth matrix
| Concern | Owner |
|--------|-------|
| title, summary, image, artist relationship | Astro content |
| catalog slug and content-derived projection | app projection layer |
| Stripe product/price IDs | D1 mapping records |
| live sellable metadata for checkout | Stripe |
| inventory and order lifecycle | D1 |

### Recommended linking model
- release pages link to canonical `/shop/[slug]/`
- shop pages never use raw Fourthwall URLs as canonical routes
- `CatalogItem` slug remains stable even if the variant mapping changes underneath

## Don't Hand-Roll

- Do not duplicate editorial content into D1.
- Do not use Stripe as the home for editorial media or prose.
- Do not expose direct browser writes to D1 or Stripe secret APIs.

## Common Pitfalls

- collapsing `CatalogItem` and `Offer` into one object too early
- storing pricing or inventory fields in Astro collections
- letting frontend routes depend on Stripe-specific IDs
- making release links point to implementation-specific backend data instead of canonical catalog slugs

## Code Examples

- static frontend consumes `CatalogItem` for browse/PDP
- frontend fetches `VariantSnapshot` from temporary adapter or Worker API without changing UI contract
- backend repositories hide D1 and Stripe implementation details behind interfaces

## Recommendation

Implement the storefront around a stable `CatalogItem` plus `VariantSnapshot` contract. Keep the browser coupled only to those contracts and route slugs. Put Stripe IDs, inventory values, and order state behind the Worker backend and D1 mappings.
