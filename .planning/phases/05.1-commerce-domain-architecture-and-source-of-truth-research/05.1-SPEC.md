# Phase 5.1 SPEC - Commerce Domain Architecture And Source-Of-Truth Research

## Deliverable

Produce a decision-complete architecture contract for the commerce domain used by Phases 6, 6.1, and 7.

## Canonical Entity Model

| Entity | Purpose | Owner | Notes |
|--------|---------|-------|-------|
| `Artist` | Editorial band/artist identity used for roster, bio, and release relationships | Astro content | Never becomes the direct sellable unit |
| `Release` | Editorial release entry tied to an `Artist` and release-specific media/copy | Astro content | May map to one or more sellable variants later |
| `DistroEntry` | Editorial distro/shop entry with merchandising copy and images | Astro content | Current `fourthwall_url` is transitional handoff wiring, not future commerce identity |
| `CatalogItem` | Storefront-facing projection consumed by browse and product detail UI | Derived by the app/backend projection layer | Built from exactly one content source: `Release` or `DistroEntry` |
| `Variant` | Sellable unit used for stock, Stripe mapping, checkout, and later order lines | Worker backend + downstream Stripe/D1 integrations | Not authored directly in Astro collections |

## Source-Of-Truth Matrix

| Concern | Authority | Rule |
|--------|-----------|------|
| Editorial copy, images, artist relationships, release metadata, distro presentation, and content slugs | Astro content | Astro remains editorial-only |
| Storefront-facing `CatalogItem` view and shop-page composition | Projection layer | Derived from authoritative sources; does not become a fourth source of truth |
| Sellable product/price configuration used for checkout | Stripe | Browser must not depend on Stripe IDs directly |
| Inventory, order lifecycle, and internal app-to-Stripe mappings | D1 | D1 remains operational state plus mapping authority |

## Locked Boundaries

- Astro content must not become the authority for pricing, stock, or checkout configuration.
- Stripe must not become the home for editorial presentation, media, or CMS copy.
- D1 must not become the source of truth for editorial content.
- `CatalogItem` is the browser-facing abstraction; `Variant` is the backend sellable abstraction underneath it.
- The projection layer bridges the sources above; it does not justify stuffing temporary commerce fields directly into the editorial collections.

## Falsifiable Requirements

1. The entity model explicitly defines `Artist`, `Release`, `DistroEntry`, `CatalogItem`, and `Variant`.
2. The source-of-truth matrix explicitly assigns ownership to Astro content, Stripe, and D1 with no ambiguous overlap.
3. Canonical rules exist for content IDs, catalog slugs, variant IDs, and backend mappings.
4. Internal backend interfaces are named and scoped clearly enough that implementation can proceed without re-deciding boundaries.
5. External Worker APIs for catalog item lookup, variant lookup, checkout-session creation, and webhook receipt are drafted.
6. The storefront-linking contract explains how release pages route into shop pages.
7. The backend naming contract explicitly uses ubiquitous-language terms and assumes TypeScript-authored contracts behind the Hono HTTP layer.
8. The API contract model explicitly uses backend-owned code-first OpenAPI with separate public/internal documents and generated frontend clients.

## Boundaries

- This phase does not implement storefront code.
- This phase does not implement D1 schema or Stripe code.
- This phase does not redesign the approved storefront UI.

## Acceptance

- A human can read the resulting docs and implement Phase 6, 6.1, and 7 without making new architecture decisions.
