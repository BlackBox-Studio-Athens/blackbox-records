# Phase 5.1 SPEC - Commerce Domain Architecture And Source-Of-Truth Research

## Deliverable

Produce a decision-complete architecture contract for the commerce domain used by Phases 6, 6.1, and 7.

## Canonical Entity Model

| Entity        | Purpose                                                                        | Owner                                              | Notes                                                                                 |
| ------------- | ------------------------------------------------------------------------------ | -------------------------------------------------- | ------------------------------------------------------------------------------------- |
| `Artist`      | Editorial band/artist identity used for roster, bio, and release relationships | Astro content                                      | Never becomes the direct sellable unit                                                |
| `Release`     | Editorial release entry tied to an `Artist` and release-specific media/copy    | Astro content                                      | May map to one or more sellable variants later                                        |
| `DistroEntry` | Editorial distro/shop entry with merchandising copy and images                 | Astro content                                      | Current `fourthwall_url` is transitional handoff wiring, not future commerce identity |
| `StoreItem`   | Storefront-facing projection consumed by browse and store item detail UI       | Derived by the app/backend projection layer        | Built from exactly one content source: `Release` or `DistroEntry`                     |
| `Variant`     | Sellable unit used for stock, Stripe mapping, checkout, and later order lines  | Worker backend + downstream Stripe/D1 integrations | Not authored directly in Astro collections                                            |

## Source-Of-Truth Matrix

| Concern                                                                                                | Authority        | Rule                                                                         |
| ------------------------------------------------------------------------------------------------------ | ---------------- | ---------------------------------------------------------------------------- |
| Editorial copy, images, artist relationships, release metadata, distro presentation, and content slugs | Astro content    | Astro remains editorial-only                                                 |
| Storefront-facing `StoreItem` view and shop-page composition                                           | Projection layer | Derived from authoritative sources; does not become a fourth source of truth |
| Sellable product/price configuration used for checkout                                                 | Stripe           | Browser must not depend on Stripe IDs directly                               |
| Stock, order lifecycle, and internal app-to-Stripe mappings                                            | D1               | D1 remains operational state plus mapping authority                          |

## Locked Boundaries

- Astro content must not become the authority for pricing, stock, or checkout configuration.
- Stripe must not become the home for editorial presentation, media, or CMS copy.
- D1 must not become the source of truth for editorial content.
- `StoreItem` is the browser-facing abstraction; `Variant` is the backend sellable abstraction underneath it.
- The projection layer bridges the sources above; it does not justify stuffing temporary commerce fields directly into the editorial collections.

## Internal Backend Contracts

- `EditorialContentSource`: reads editorial content needed to project shop-facing data.
- `StoreItemProjector`: derives a canonical `StoreItem` from one content source.
- `StoreItemReader`: resolves browse/detail `StoreItem` views for the storefront.
- `ItemAvailabilityReader`: resolves storefront-facing variant state without leaking Stripe or D1 shapes.
- `StoreItemOptionRepository`: stores backend mappings between content-source identities, store item slugs, and variant identities.
- `VariantStripeMappingRepository`: stores backend mappings from canonical `VariantId` values to Stripe product/price identities.
- `ItemAvailabilityRepository`: owns stock and later availability reads/writes behind the backend boundary.
- `CreateCheckoutSession`: backend use case that creates Stripe-backed checkout sessions from canonical app identities.
- `ReceiveStripeWebhook`: backend entry point for Stripe webhook receipt and later order/stock side effects.

## External Worker API Contract Intent

- Public shopper API:
  - `GET /api/store/items`
  - `GET /api/store/items/:storeItemSlug`
  - `GET /api/store/items/:storeItemSlug/variants`
  - `POST /api/checkout/sessions`
- Stripe integration API:
  - `POST /api/stripe/webhooks`
- Public shopper routes belong in the public OpenAPI document and generated frontend client package.
- Future staff-only stock APIs belong in the internal OpenAPI document introduced later by Phase `06.1.1`.

## Canonical IDs And Linking Rules

- `ArtistId`, `ReleaseId`, and `DistroEntryId` are the Astro content entry identities already owned by the editorial collections.
- `StoreItemSlug` is the canonical shopper-facing shop identity used by routes and frontend/backend contracts.
- `VariantId` is the backend-owned sellable identity used for stock, Stripe mappings, checkout, and order lines.
- `VariantId` must not equal or expose raw Stripe product IDs or price IDs.
- Stripe IDs live behind backend mapping records and never become frontend routing or state identities.
- Release and distro entry points must resolve to canonical native shop routes using `StoreItemSlug`, not raw `fourthwall_url` values or backend persistence IDs.

## Falsifiable Requirements

1. The entity model explicitly defines `Artist`, `Release`, `DistroEntry`, `StoreItem`, and `Variant`.
2. The source-of-truth matrix explicitly assigns ownership to Astro content, Stripe, and D1 with no ambiguous overlap.
3. Canonical rules exist for content IDs, store item slugs, variant IDs, and backend mappings.
4. Internal backend interfaces are named and scoped clearly enough that implementation can proceed without re-deciding boundaries.
5. External Worker APIs for store item lookup, variant lookup, StartCheckout, and webhook receipt are drafted.
6. The storefront-linking contract explains how release pages route into shop pages.
7. The backend naming contract explicitly uses ubiquitous-language terms and assumes TypeScript-authored contracts behind the Hono HTTP layer.
8. The API contract model explicitly uses backend-owned code-first OpenAPI with separate public/internal documents and generated frontend clients.

- This phase does not implement D1 schema or Stripe code.
- This phase does not redesign the approved storefront UI.
