# 18-01 Plan - Audit Valibot Usage And Lock Zod As The Validator Standard

## Objective

Remove any direct Valibot usage from repo-authored code and document Zod as the validation standard while being honest about unavoidable Prisma transitive dependencies.

## Tasks

1. Audit dependency and import surfaces.
   - Search source, tests, scripts, package manifests, generated-contract sources, and planning docs.
   - Run `pnpm why valibot --recursive`.
   - Classify each hit as direct app usage, generated/transitive dependency evidence, or historical planning text.

2. Remove or replace direct usage.
   - If any direct `valibot` package manifest entry exists, remove it.
   - If any direct source import exists, replace with equivalent Zod schema.
   - Add focused tests for any changed runtime boundary.

3. Clean small Zod deprecations if scoped.
   - If Astro/Zod `.url()` or `.email()` hints are still present and the Zod 4 replacement is straightforward, update them behavior-preservingly.
   - Keep this secondary to Valibot removal; do not redesign content schemas.

4. Document the validator policy.
   - Update `.planning/codebase/STACK.md` or a nearby codebase convention doc to say repo-authored validation uses Zod.
   - Record that Valibot may remain in `pnpm-lock.yaml` only as a Prisma transitive dependency.

5. Record validation evidence in `18-VALIDATION.md`.

## Verification

- `rg -n "from 'valibot'|from \"valibot\"|valibot" package.json apps packages scripts .planning`
- `pnpm why valibot --recursive`
- Targeted schema tests if code changed.
- `pnpm test:unit`
- `pnpm check`
- `pnpm build`

## Stop Conditions

- Stop before replacing Prisma only to remove transitive Valibot.
- Stop before moving OpenAPI contracts away from Zod.
- Stop before adding ArkType or Valibot as an alternate validator.
