# 07-12 Plan - Treat All Current Distro And Release Entries As Sellable Store Candidates

## Objective

Make the storefront/catalog discovery model match the label reality: current distro entries and current releases are real sellable item candidates, even when quantities are not yet counted.

## Scope Boundary

This commit updates the item candidacy and linking model only. It does not add checkout stock authority, real Stripe Price mappings, webhooks, order state, shipping, or production cutover.

## Tasks

1. Update store-item discovery so every current distro JSON entry remains a store candidate.
2. Update release discovery so every current release Markdown entry can become a native store candidate, even if it still has legacy `merch_url` or external merch metadata.
3. Preserve compatibility redirects/fallbacks for old release and distro routes where needed.
4. Keep shopper-facing canonical URLs based on sellable item option language, not legacy release shorthand.
5. Document that item candidacy does not mean counted stock or Stripe test readiness.

## Verification

- all current distro entries are visible store candidates.
- all current releases are visible store candidates or have explicit native aliases.
- legacy external merch metadata no longer blocks native local checkout readiness.
- canonical item-option URLs still describe the purchased item option.
- `pnpm test:unit`, `pnpm check`, and `pnpm build` pass.

## Explicit Exclusions

- No real inventory counts.
- No Stripe test Price IDs.
- No local fake stock yet.
- No webhook, order, or shipping behavior.
