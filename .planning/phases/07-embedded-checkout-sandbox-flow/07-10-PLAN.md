# 07-10 Plan - Add Checkout Return And Retry State UI Through ReadCheckoutState

## Objective

Add static return/retry UI that reads checkout state from the Worker instead of trusting browser query parameters.

## Status

Completed 2026-04-25. The static return route now reads `session_id` only as input to Worker-owned `ReadCheckoutState`, renders app-owned checkout return states, and offers retry/cart/item/store actions without creating browser payment authority.

## Scope Boundary

Return pages are shopper feedback only. They do not mark orders paid, decrement stock, or create order authority.

## Tasks

1. Add the static checkout return route/state for `/store/[slug]/checkout/return`.
2. Read `session_id` only as an input to Worker-owned `ReadCheckoutState`.
3. Render paid, open, processing, expired, and unknown states using app-owned labels.
4. Offer retry/back-to-item/back-to-cart actions without duplicating checkout authority in the browser.
5. Keep the return page aligned with the new item-option URL and cart/summary language.

## Verification

- return UI calls the Worker state endpoint.
- raw Stripe query params are not treated as payment truth.
- retry path routes back through the canonical checkout shell.
- cart state does not become payment authority.
- `pnpm test:unit`, `pnpm check`, and `pnpm build` pass.

## Explicit Exclusions

- No webhook state mutation.
- No D1 order writes.
- No stock changes.
- No shipping flow.
