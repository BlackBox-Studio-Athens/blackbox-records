# Phase 7.1 Research - Cloudflare Pages Static Frontend Migration

## Sources Checked

- Cloudflare Pages Direct Upload docs: https://developers.cloudflare.com/pages/get-started/direct-upload/
- Cloudflare Pages Direct Upload with CI docs: https://developers.cloudflare.com/pages/how-to/use-direct-upload-with-continuous-integration/
- Cloudflare Pages monorepo docs: https://developers.cloudflare.com/pages/configuration/monorepos/
- Cloudflare Pages Astro guide: https://developers.cloudflare.com/pages/framework-guides/deploy-an-astro-site/

## Findings

Cloudflare Pages can host the current Astro static output without changing the frontend runtime model. The key decision is whether Cloudflare runs the build directly from the connected repository or GitHub Actions uploads prebuilt assets.

For this repo, Direct Upload from GitHub Actions is the safer default because:

- the existing Pages workflow already runs `pnpm test:unit`, `pnpm check`, and `pnpm build`
- the repo uses a pnpm monorepo and Astro app under `apps/web`
- the Worker backend has separate Cloudflare deployment concerns that should not be blended into a Pages Functions deployment
- the migration can preserve one CI-owned artifact boundary: `apps/web/dist`

Cloudflare Pages monorepo support is relevant if using Cloudflare's own Git integration, but Direct Upload avoids re-creating all CI build logic in the Cloudflare dashboard and keeps reviewable workflow changes in git.

## Risks

- Checkout return origins can break if the Worker allowlist is not updated for Cloudflare Pages preview and production origins.
- Public env vars can drift if Pages project variables are configured manually and not documented.
- The `/stock/` static UI can become publicly reachable if served on the public Pages hostname; production-safe operator use still requires the protected ops hostname and Cloudflare Access.
- Keeping GitHub Pages and Cloudflare Pages active simultaneously can confuse canonical URL, sitemap, and SEO behavior unless docs and deployment settings are explicit.
- A custom domain cutover can create DNS/cache ambiguity if done before preview and production branch deploys are verified.

## Recommended Approach

Use a five-plan phase:

1. Lock the deployment contract and rollback posture.
2. Add Pages project config and a direct-upload CI workflow.
3. Align browser env and Worker return-origin contracts.
4. Validate preview/production branch deploys and Worker API routing.
5. Retire GitHub Pages as canonical in docs after acceptance.

Do not remove the GitHub Pages workflow until Cloudflare Pages has passed acceptance. Do not add Pages Functions or move backend routes into the frontend host.
