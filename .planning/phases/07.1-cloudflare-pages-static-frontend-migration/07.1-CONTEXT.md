# Phase 7.1: Cloudflare Pages Static Frontend Migration - Context

**Gathered:** 2026-04-24
**Status:** Planning complete

<domain>
## Phase Boundary

Phase 7.1 moves the static Astro frontend deployment target from GitHub Pages to Cloudflare Pages while keeping the frontend static, the Worker backend separate, and runtime business secrets server-only.

This phase is inserted after Phase 7 and before Phase 8 because checkout browser routes and return-origin behavior should be stable before webhooks, order state, shipping, and sandbox release-gate verification depend on the final static hosting origin.

</domain>

<decisions>
## Implementation Decisions

- **D-01:** Cloudflare Pages becomes the canonical static frontend host only after preview and production-branch validation pass.
- **D-02:** GitHub Pages remains a rollback/safety reference during the migration; removal or deactivation is a final plan, not a first step.
- **D-03:** The frontend remains `output: "static"` Astro. Do not introduce SSR, Pages Functions, or a Pages-hosted backend in this phase.
- **D-04:** The Worker remains the backend/BFF for Stripe, D1, internal stock APIs, webhooks, and future BOX NOW work.
- **D-05:** Prefer Cloudflare Pages Direct Upload from GitHub Actions with Wrangler so CI keeps running the repo-owned `pnpm test:unit`, `pnpm check`, and `pnpm build` gates before deploy.
- **D-06:** Cloudflare Pages browser variables must stay public-only: `PUBLIC_BACKEND_BASE_URL`, `PUBLIC_STRIPE_PUBLISHABLE_KEY`, and optional `PUBLIC_CHECKOUT_CLIENT_MODE` for non-production local/mock cases.
- **D-07:** Runtime business secrets remain Worker secrets or D1 bindings. Do not move `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, D1 credentials, BOX NOW credentials, or Cloudflare Access policy secrets into Pages env vars.
- **D-08:** Checkout return origins and CORS/routing docs must be updated so the Worker accepts the Cloudflare Pages preview/production origins intentionally, not through broad wildcard allowances.
- **D-09:** The protected ops UI at `/stock/` is still static Astro code, but it is production-safe only when served through the Cloudflare Access protected ops hostname with `/api/internal/*` routed to the Worker.

</decisions>

<research>
## Current Cloudflare Docs Readback

- Cloudflare Pages supports monorepos and lets each Pages project define a root directory/build configuration.
- Cloudflare Pages Direct Upload supports deploying prebuilt assets with Wrangler from CI, which fits this repo because GitHub Actions already owns build/test gates.
- Cloudflare's Astro Pages guide still treats Astro as a supported static framework and documents standard build-output deployment.

</research>

<specifics>
## Specific Ideas

- Introduce a Pages project for the static web app, separate from the Worker backend project.
- Keep `apps/web/dist` as the artifact boundary for Pages deploys.
- Keep current GitHub Pages workflow present until the acceptance plan explicitly marks Cloudflare Pages canonical.
- Add Cloudflare Pages CI credentials separately from Worker deployment credentials only if Cloudflare requires a narrower token; otherwise document exact token permissions.
- Validate both public storefront routes and hidden `/stock/` static route generation under Pages before changing canonical docs.

</specifics>
