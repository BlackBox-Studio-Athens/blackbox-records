# 07.1-01 Plan - Lock The Cloudflare Pages Deployment Contract And Rollback Posture

## Objective

Freeze the target deployment model before changing CI: Cloudflare Pages becomes the future canonical static frontend host, GitHub Pages remains rollback until acceptance, and the Worker backend remains separate.

## Scope Boundary

Planning/docs and config contract only. Do not add a workflow yet, do not delete GitHub Pages, and do not change runtime app behavior.

## Tasks

1. Document the Cloudflare Pages project intent, static artifact boundary, and rollback posture.
2. Lock that `apps/web/dist` remains the deploy artifact.
3. Lock that Pages must not own backend routes, Stripe secrets, D1 access, or operator auth.
4. Identify required Cloudflare Pages CI credentials and project naming conventions without committing account-specific values.
5. Update README, AGENTS, and phase context with the deployment boundary.

## Verification

- Docs clearly distinguish Cloudflare Pages static hosting from the Worker backend.
- GitHub Pages remains documented as rollback during migration.
- No runtime files, workflows, or app behavior change.
- `git diff --check` passes.

## Explicit Exclusions

- No Pages workflow.
- No GitHub Pages removal.
- No DNS/custom-domain cutover.
- No Pages Functions.
- No production Stripe changes.
