# 08-03.1 Plan - Add Stripe-Mock API Local Checkout Simulation Harness

## Objective

Add an optional local stack path that points the Worker Stripe SDK at the official `stripe/stripe-mock` API server while keeping the existing no-Docker in-process mock stack as the default.

## Scope Boundary

This commit validates local Stripe API request shape and signed webhook fixture handling without real Stripe account access. It does not replace real Stripe testmode validation, and it does not add paid-order transitions, stock decrement, production config, or account-specific Stripe values.

## Tasks

1. Add an optional `stripe-mock-api` local stack mode that requires official `stripe-mock` on `127.0.0.1:12111`.
2. Run local D1 preparation and all-items mock commerce seeding before starting the stack.
3. Start the Worker with `STRIPE_API_BASE_URL=http://127.0.0.1:12111` and a local fake test secret.
4. Keep the static frontend in mock checkout panel mode so the browser never depends on real Stripe.js.
5. Add a webhook simulator script that posts signed checkout-session fixture events to `/api/stripe/webhooks`.
6. Keep `pnpm dev:stack:stripe-mock` as the canonical default no-Docker local stack.
7. Keep mock-only behavior in dev scripts, test helpers, local Wrangler env config, or the existing frontend mock panel rather than production checkout/order use cases.

## Verification

- stack planning tests prove the optional mode uses D1 prep, local mock commerce seed, Stripe API base URL override, and frontend mock checkout mode
- startup fails clearly when official `stripe-mock` is not listening
- gateway tests prove the Stripe SDK can target a local API base URL
- official `stripe-mock` mode exercises the real Stripe SDK request path rather than a production-code fake response
- webhook simulator tests prove signed fixture event construction without real webhook secrets
- Browser Use smoke can exercise PDP/cart/checkout into the mock panel, then acknowledge a signed webhook fixture

## Explicit Exclusions

- No real Stripe keys, Price IDs, products, webhook secrets, or sandbox writes
- No OpenAPI or generated API client changes
- No order lifecycle mutation
- No stock decrement
- No real embedded Checkout browser validation
