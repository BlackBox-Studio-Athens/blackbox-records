# Phase 6.1 Research - Worker Commerce State Foundation

## Standard Stack

- D1 for operational state and mappings
- Prisma for runtime access
- Wrangler-applied SQL migrations
- Worker repositories hiding storage details from the frontend

## Architecture Patterns

- backend repositories behind stable frontend-facing contracts
- D1 state separate from editorial content
- mappings layer between app identities and Stripe identities

## Don't Hand-Roll

- do not duplicate editorial content into D1
- do not bypass repositories with ad hoc direct DB reads
- do not treat Prisma schema as the only migration truth on D1

## Common Pitfalls

- coupling frontend types directly to D1 tables
- letting D1 become the source of truth for editorial content
- introducing price authority into D1 before Stripe integration is defined

## Code Examples

- Worker repository returns `ItemAvailability`
- Worker mapping repository resolves `StoreItem` or `Offer` to Stripe IDs
- migrations are generated via Prisma diff and applied via Wrangler
