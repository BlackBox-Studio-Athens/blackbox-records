# Phase 6.1: Worker Commerce State Foundation - Context

**Gathered:** 2026-04-20
**Status:** Planning complete

<domain>
## Phase Boundary

Phase 6.1 introduces D1 and Prisma inside the separate Worker backend after the static storefront contract already exists. It does not introduce Stripe checkout, webhook semantics, or the full order lifecycle implementation from Phase 8.

</domain>

<decisions>
## Implementation Decisions

### Backend database scope
- **D-01:** Phase 6.1 bootstraps local D1 and binding names for the Worker backend.
- **D-02:** Prisma is used for runtime access on D1, and migrations remain SQL applied through Wrangler.
- **D-03:** The first D1-backed reads are limited to backend commerce-state concerns needed behind Worker repositories and APIs.
- **D-04:** Full order lifecycle and authoritative paid-state semantics remain deferred to Phase 8.

### Repository and API boundary
- **D-05:** The Phase 6 storefront contract must not change in this phase.
- **D-06:** Backend repositories may begin moving temporary offer or mapping reads from static/fixture data to D1-backed data where useful.
- **D-07:** Price authority is still not D1's responsibility; Stripe remains the later authority for sellable commerce data.

### Phase shape
- **D-08:** This phase is about the separate backend, not the frontend.
- **D-09:** D1 and Prisma are introduced behind Worker repositories and backend API seams.
- **D-10:** Phase 7 depends on this phase so checkout work starts only after backend state foundations exist.

</decisions>

<specifics>
## Specific Ideas

- Keep D1 focused on operational state and mappings, not editorial duplication.
- Keep the storefront contract stable and treat D1 as one backend source behind that contract.
- Make Phase 6.1 the one place where D1 + Prisma assumptions become concrete before Stripe arrives.

</specifics>
