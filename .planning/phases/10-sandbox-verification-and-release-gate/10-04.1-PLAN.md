# 10-04.1 Plan - Add Worker-Owned Native Checkout Feature Gate

## Objective

Add a Worker-owned runtime feature gate for native checkout so operators can disable checkout without redeploying once
Cloudflare Flagship is configured.

## Scope Boundary

This is a checkout capability gate, not a replacement for separate Worker environments. Sandbox and production
environments still isolate D1 data, secrets, webhook endpoints, return origins, and release evidence.

## Tasks

1. Add a backend feature flag seam using OpenFeature-compatible Cloudflare Flagship as the maintained provider target.
2. Add first flag `native_checkout_enabled`, evaluated by the Worker before Stripe Checkout Session creation or pending
   order creation.
3. Fail closed for sandbox and production when the Flagship binding is missing or evaluation fails; keep local/mock
   checkout enabled by default for the no-account local stack.
4. Add a browser-safe public capabilities read that exposes only whether native checkout is currently enabled.
5. Update checkout UI to disable the payment action when Worker capabilities say native checkout is unavailable.
6. Document the Cloudflare setup contract: binding name `FLAGS`, flag key `native_checkout_enabled`, and no committed
   Flagship app ID until the app exists and the non-secret ID is explicitly approved.

## Verification

- feature reader defaults are tested
- disabled `StartCheckout` returns `503` and does not call Stripe or order repositories
- public capabilities route exposes only browser-safe state
- checkout UI blocks payment when capabilities disable native checkout
- generated OpenAPI/client artifacts are updated
- no-secret commerce boundary audit still passes

## Explicit Exclusions

- No real Stripe keys, Price IDs, webhook secrets, or Stripe account setup
- No BOX NOW credentials or portal setup
- No production D1 mutation or production deployment
- No Cloudflare Flagship app ID in committed Wrangler config
- No browser-owned flag authority
