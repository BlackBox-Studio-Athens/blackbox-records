# 13-01 Plan - Configure And Wire Dynamic Payment Methods

## Objective

Enable Stripe dynamic payment methods through the named `BlackBox merch checkout` Payment Method Configuration while
making missing `STRIPE_PAYMENT_METHOD_CONFIGURATION_ID` an invalid application state and keeping CI verification API-based.

## Must Haves

- Checkout Session creation must send `payment_method_configuration` and must not send `payment_method_types`.
- `createStripeCheckoutGateway()` must throw before Checkout Session creation when
  `STRIPE_PAYMENT_METHOD_CONFIGURATION_ID` is absent.
- Every local/mock/sandbox/production Worker mode must have a `STRIPE_PAYMENT_METHOD_CONFIGURATION_ID` binding value.
- Stripe configuration automation must be dry-run by default and mutate only with `--apply`.
- Stripe configuration evidence must be planning-safe, redacted, and suitable for `13-VALIDATION.md`.
- The CI gate must verify code invariants and Stripe test-mode configuration by API when Stripe test credentials are
  configured; hosted Checkout browser inspection is not the Phase 13 CI gate.
- Existing card-focused sandbox smoke scenarios must remain deterministic.

## Decision Coverage

- **D-01:** Production Checkout uses Stripe dynamic payment methods through a named Payment Method Configuration.
- **D-02:** Allowed buyer-visible methods are card rails, Apple Pay, Google Pay, and Link.
- **D-03:** PayPal must never be displayed.
- **D-04:** Klarna and all BNPL-style methods must never be displayed.
- **D-05:** Bank-debit, bank-transfer, and mandate-style methods must never be displayed.
- **D-06:** The implementation verifies the actual Stripe configuration response; banned method categories are policy
  groups, not a complete static enum.
- **D-07:** A repo script is the canonical Stripe configuration automation surface.
- **D-08:** The script is dry-run by default.
- **D-09:** The script mutates Stripe configuration only with `--apply`.
- **D-10:** Allowed configurable methods are `card`, `apple_pay`, `google_pay`, and `link`.
- **D-11:** Configurable banned methods include `paypal`, `klarna`, known BNPL methods, and bank-debit or mandate-style
  methods returned by Stripe.
- **D-12:** Banned methods pass only when absent, unavailable, or effectively off; effectively-on banned methods fail.
- **D-13:** Dashboard-only or account-eligibility gaps are reported instead of silently passed.
- **D-14:** Required script output is a redacted, planning-safe human summary.
- **D-15:** Payment Method Configuration IDs in evidence show object prefix plus last four characters only.
- **D-16:** Evidence records returned Stripe method keys and statuses only.
- **D-17:** Phase 13 implementation evidence belongs in `13-VALIDATION.md`.
- **D-18:** `STRIPE_PAYMENT_METHOD_CONFIGURATION_ID` is required for the application to run in Phase 13 checkout mode.
- **D-19:** Missing configuration fails at `createStripeCheckoutGateway()` / gateway construction.
- **D-20:** Local stripe-mock, local stripe-test, sandbox, and production modes all require the binding.
- **D-21:** Gateway/factory tests prove missing binding failure and configured `payment_method_configuration` payloads.
- **D-22:** Checkout Session creation removes hardcoded `payment_method_types: ['card']`.
- **D-23:** Browser code never receives Payment Method Configuration IDs, provider names, Stripe IDs, D1 bindings, or
  internal feature-gate details.
- **D-24:** CI verifies Stripe configuration through the Stripe API/configuration response, not hosted Checkout browser
  inspection.
- **D-25:** Normal CI proves checkout payload and missing-config code invariants.
- **D-26:** A Stripe-backed CI job verifies test-mode Payment Method Configuration by API.
- **D-27:** Browser Use hosted Checkout validation is optional go-live sanity evidence, not the Phase 13 CI gate.
- **D-28:** Live-mode API verification repeats later as a go-live gate after live readiness exists.
- **D-29:** Existing deterministic card-based sandbox smoke coverage remains intact.

## Threat Model

<threat_model>

| Threat                                                                          | Severity | Mitigation                                                                                                            |
| ------------------------------------------------------------------------------- | -------- | --------------------------------------------------------------------------------------------------------------------- |
| A banned payment method is enabled in Stripe and appears to buyers.             | high     | The Stripe configuration script classifies returned methods and fails if a returned banned method is effectively on.  |
| A Checkout Session silently falls back to card-only or Stripe account defaults. | high     | The gateway factory requires `STRIPE_PAYMENT_METHOD_CONFIGURATION_ID`, and tests assert payload shape.                |
| Account-specific Stripe IDs or secrets leak into committed docs.                | medium   | Evidence prints only redacted summaries with object prefix plus last four characters; full IDs remain runtime config. |
| Browser code learns provider/config details.                                    | medium   | Payment Method Configuration ID stays in Worker bindings and is never added to Astro public env or API responses.     |

</threat_model>
